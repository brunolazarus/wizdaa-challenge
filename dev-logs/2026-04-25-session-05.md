### Key Design Decisions Made

**`BalanceFreshnessIndicator` receives `isStale: boolean` as a prop (not `asOf` alone):**
The component is in `components/ui/` — primitive and stateless. Computing staleness from `asOf` inline would require importing `MANAGER_STALENESS_THRESHOLD_MS` from a hooks file, which is the wrong direction for a UI primitive. Instead, `RequestCard` (the feature layer) calls `isBalanceStale(balance?.asOf)` and passes the result down. Keeps the threshold logic in one place (`use-manager-actions.ts`).

**Approve/Deny buttons blocked at the UI level while balance is stale or fetching:**
`useBalance` has `staleTime: 0`, so it always re-fetches on mount. When `RequestCard` mounts, balance starts in `isFetching: true` and the buttons are disabled. Once HCM responds with a fresh `asOf` (within 60s), they enable. This gives the manager honest UI state before acting. `ensureFreshBalance` inside the mutation hooks serves as a second safety net — it re-fetches server-side if somehow stale data slipped through.

**`RequestCard` owns `settled` state locally:**
After approve or deny succeeds, the card replaces the action buttons with a confirmation badge (`Approved` / `Denied`). This is local ephemeral UI state that belongs in the card, not in the query cache. The cache invalidation (`queryKeys.requests()`) is handled by the mutation hooks — the list will re-filter on the next render, removing settled cards from `PendingRequestList`.

**Storybook interaction tests for the two required manager paths:**
- `ApproveConfirmed`: wait for balance load → click Approve → `findByText('Approved')`
- `DenyConfirmed`: wait for balance load → click Deny → `findByText('Denied')`
- `ApprovingInFlight`: click Approve → MSW delays forever → assert `Approving…` button text
