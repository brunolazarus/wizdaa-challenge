### 2. TRD (Technical Requirement Document)

Written collaboratively via conversation before any code. Lives at `docs/TRD.md`.

**Decisions locked:**

| Decision | Choice | Rationale |
|---|---|---|
| Optimistic update strategy | "Pending", not "Approved" | Legal/liability exposure if employee sees "approved" then gets denied |
| CAP theorem stance | CP (Consistency over Availability) | Wrong answer worse than slow answer in HR context |
| DB design implication | Append-only event log, not mutable rows | Auditable, rollback-safe, status never overwritten |
| Cache invalidation scope | Per-mutation, per `[employeeId, locationId]` key | Surgical — unrelated balances stay warm |
| Read-after-write verification | `version: number` field must increment | HCM 200 is not sufficient — must verify write actually applied |
| Background refresh guard | `useIsMutating` suppresses poll results during in-flight mutation | Prevents poll from clobbering optimistic state |
| Background polling interval | 30 seconds | Frequent enough for HCM-side changes, low server cost |
| State management | React Query cache + Context API (minority global) + useState (local) | No Redux/Zustand for server data |
| Initial hydration | Server Component + `HydrationBoundary` | No loading spinner on first paint |
| Test strategy | Vitest (unit) + Storybook (visual/docs/interaction) + Vitest+MSW (integration) | Strict layer separation |
| Event-driven architecture | Rejected for now — named as upgrade path | HCM is request/response only; polling + guard achieves same correctness |
| Anniversary timer | Opt-in via `HCM_ANNIVERSARY_TIMER_ENABLED=true` | Dev/demo tool only; tests use manual endpoint |

**Required tests locked in TRD §2.4 (must not be removed):**
- Background refresh must not clobber in-flight optimistic state
- Background refresh lands cleanly with no mutation in-flight → indicator shown
