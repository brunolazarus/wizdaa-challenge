### Key Design Decisions Made

**Failure injection via header takes total precedence over random rates:**
```ts
function shouldFail(forceFailure, type, rate) {
  if (forceFailure !== null) return forceFailure === type  // deterministic
  return Math.random() < rate                              // random in dev
}
```
Side effect: passing any non-matching `forceFailure` (e.g. `'conflict'` to a read)
disables ALL random checks for that call. Tests use this to get deterministic reads
without a separate bypass mechanism.

**`WriteResult` discriminated union — silent failure is a first-class return type:**
```
{ type: 'success' }   — write applied, version incremented
{ type: 'silent' }    — 200 but version unchanged — client must verify
{ type: 'conflict' }  — 409
{ type: 'timeout' }   — delay 10s, 504
{ type: 'rejected' }  — 422 (insufficient balance or not found)
```
Both `success` and `silent` return HTTP 200 — the client's read-after-write
version check is the only way to distinguish them.

**Store reads in tests bypass `getBalance`'s randomness:**
Tests that assert exact stored values use `getStore().get(storeKey(...))` directly.
Only tests specifically targeting `getBalance`'s random behaviour go through the function.

**Anniversary timer guards:**
- Only runs in Node (`typeof process !== 'undefined'`)
- Only when `HCM_ANNIVERSARY_TIMER_ENABLED=true`
- Uses `globalThis.__hcmAnniversaryTimer` to prevent duplicate timers across hot reloads
- CI workflow sets `HCM_ANNIVERSARY_TIMER_ENABLED: 'false'` explicitly
